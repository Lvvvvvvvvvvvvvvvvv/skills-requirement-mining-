# 🔍 Requirement Mining — 显性需求转隐性需求

> 一个专为 AI/SaaS 产品经理设计的 Claude Skill，帮你穿透客户的表面表述，快速还原真实的隐性需求。

---

## 为什么需要这个 Skill？

客户说："能不能加个一键导出？"
他真正的意思可能是："我不想在关键汇报前被系统卡住。"

产品经理最难的工作之一，不是评审需求，而是**在客户开口的那一刻判断他真正想要什么**。客户描述的永远是解决方案，而不是问题本身。把"显性需求"转化为"隐性需求"，是做出正确产品决策的前提——但这个过程耗时、依赖经验、容易跑偏。

这个 Skill 将这套推导过程结构化，帮你在几秒内完成原本需要思考半小时的分析。

---

## 适用人群

- AI/SaaS 产品经理
- 需求分析师
- 售前/解决方案架构师
- 任何需要对接客户需求并转化为产品语言的角色

## 核心能力

| 能力 | 说明 |
|------|------|
| 🧠 六维推导框架 | 从用户身份、表层需求、触发动机、痛点溯源、真实目标到隐性需求，逐层剥离 |
| 🎯 信息充分性判断 | 输入信息不足时，自动追问 1～2 个关键问题，而不是胡乱推导 |
| 📋 结构化输出 | 推导链每步严格一句话 + 隐性需求一句话概括，清晰可直接用于 PRD |
| 🏷️ 隐性需求分类 | 内置 AI/SaaS 场景常见隐性需求类型（控制感/信任感/降低认知负荷等） |
| 👥 多角色冲突识别 | 自动识别"提需求的人"和"用需求的人"不一致的情况 |

## 输出示例

**输入：**
> 我们大客户的 HR 总监说，系统里的考勤报表导出太麻烦了，每次要点很多步，能不能先做个一键导出？

**输出：**

```
【推导过程】
① 用户身份：HR 总监，管理决策者而非日常操作者，感知来自下属反馈或亲身使用时的挫败感
② 表层需求：将考勤报表导出流程从多步操作简化为一键完成
③ 触发动机：有明确的对外汇报或合规节点，在关键时刻被导出流程卡住，且不止一次
④ 痛点溯源：不是嫌步骤多，而是每次都要依赖特定操作路径，有出错风险且时间不可控
⑤ 真实目标：在需要数据的关键时刻立刻拿到结果，维持对下属和上级的掌控感

【隐性需求】
"在需要考勤数据的关键时刻，能不依赖任何操作记忆立刻拿到想要的结果。"
```

---

## 推导框架说明

### 六维推导链

```
显性需求（客户原话）
       ↓
① 用户身份    → 他是谁？决策者还是执行者？
       ↓
② 表层需求    → 剥离情绪后，他字面上要什么？
       ↓
③ 触发动机    → 什么事让他现在提这个需求？
       ↓
④ 痛点溯源    → 现状哪个环节让他不满？
       ↓
⑤ 真实目标    → 他想达成的业务/个人目标是什么？
       ↓
⑥ 隐性需求    → 去掉解决方案外壳，本质诉求是什么？
```

### AI/SaaS 场景隐性需求速查

常见隐性需求大多落在以下五类：

| 类型 | 表现特征 | 典型关键词 |
|------|----------|------------|
| **控制感** | 对流程/数据有掌控和可见性 | "我想随时看到"、"让我知道进度" |
| **信任感** | 系统/结果可信、可解释 | "我不确定这个数对不对"、"AI 的结果我不敢用" |
| **减少责任** | 出了问题有据可查、有人可问 | "万一出错谁负责"、"要留记录" |
| **降低认知负荷** | 少思考、少记忆、少操作 | "太复杂了"、"我每次都要想半天" |
| **彰显价值** | 让上级/同事看到自己的贡献 | "能不能有个报告"、"我想展示一下效果" |

## 安装方式

### 方式一：导入 `.skill` 文件（推荐）

1. 下载本仓库中的 `requirement-mining.skill` 文件
2. 打开 Claude 客户端 → 设置 → Skills → Import Skill
3. 选择下载的 `.skill` 文件，导入完成

### 方式二：手动添加

1. 下载本仓库
2. 将 `requirement-mining/` 文件夹放入你的 Skills 目录
3. 重启 Claude 客户端使其生效

---

## 使用方式

安装后，直接在对话中描述客户的需求原话即可，无需特殊指令。

**触发关键词示例：**
- "客户说要 xxx"
- "帮我分析这个需求"
- "这个需求背后是什么"
- "挖掘隐性需求"
- "客户提了一个需求：..."

**如果输入信息不足**，Skill 会自动追问 1～2 个关键问题（例如"这个需求是谁提的？"），补充后立即开始推导，不会反复盘问。

---

## 使用建议

**✅ 适合在以下时机使用：**
- 客户访谈结束后，整理需求时
- 需求评审前，判断一个需求是否值得做
- PRD 撰写前，确认"问题定义"部分
- 收到紧急需求时，判断真正的优先级

**⚠️ 注意事项：**
- 输入客户原话效果最好，不要先自己加工再输入
- 如果你已知客户的角色和背景，可以一并提供，推导会更准确
- 隐性需求是"本质诉求"，不是功能方向，请勿将其直接复制进 PRD 的功能描述
